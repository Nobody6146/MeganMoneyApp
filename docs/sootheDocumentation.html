Soothe Documentation

The soothing way to design dynamic webpages!


Introduction

What is Soothe?

A light-weight and easy frontend library for making dynamic html webpages and client side applications likes SPAs

Because SootheApp is a library and not a framework, it can be dynamically added to any new or existing HTML page without side effects.

It's modular system allows you to select and combine the functionality you need to acheive your desired end result. This means many things can be done several ways, and you are given the building blocks to custom tailor each solution.

SootheApp handles these 3 key features

  • 1. Model binding
  • 2. DOM manipulation
  • 3. Client-side routing

Design Goals

Making dynamic webpages should be easy and hassle free. We shouldn't have to write complex JS queries and code to manipulate our HTML or push data in and out of it.

There is a strong seperation of concerns. Let's worry about writting/designing our page in the HTML and CSS files only! Likewise, we should only be writting our JavaScript in JS files and not mixing the two!

SootheApp's models binding act as a communication layer/pipeline between our UI and app logic layers. These modals are global where anything can read or write to and acts as a shared contract between contributors. Everyone is reading/writing data as the same format, this makes communication easy!

NOTE: Models don't have to be used to communicate with app logic. Models can also be used exclusively to store individual element state or communicate with other DOM elements

What makes Soothe different?

There are many popular front-end frameworks, but Soothe is unique with it's model binding and event based framework. Here are a few of the differences between Soothe and other popular frameworks

  • 1. The most popular frameworks like React or Angular require running a backend server or views/components to be compiled. SootheApp is a client-side library which means no view building or backend is needed. This means Soothe applications can actually be served over static filesystems.
  • 2. Many of the client-side libraries such as Vue require knowing model state and HTML nodes to be bound to during framework initiation. Soothe app is 100% dynamic and any part of the ecosystem can be attached at any time and things will work. This means:
    • a. Whenever an HTML element is added to the DOM, or is changed, it can immmediately trigger binding events and start responding to them
    • b. Whenever a JS callback is added, it will be immediately ready to start handling callbacks when events trigger
    • c. Whenever new models or model properties are bound, they will immediately trigger events
    • NOTE: This makes SootheApp really development friendly as you can easily use developer tools in browser to manipulate DOM and JS models to instally test changes without even reloading pages! NOTE 2: This also means we can easily do custom HTML displays that get determined at runtime!
  • 3. Many libraries can render ugly HTML documents with unwanted string/characters being present if the developer wrote bad code. Because SootheApp only uses standard html elements and custom attributes, no such side-effects exists. Incorrect coding is invisble to the UI.
  • 4. Many frameworks don't support 2-way model binding out of the box. SootheApp does.

How does DOM manipulation and model binding work?

Developers will determine a data structure for their application state and communication. This data is then bound to the Soothe framework into a model. Any element via the DOM or JavaScript can read and write to this model.

Changes to these models are automatically tracked by the framework and events are dispatched from the updates. From here, HTML elements and JavaScript callback functions can subscribe to these events and respond to them.

HTML elements can spawn in the ways below

  • 1. Setting HTML element attributes/properties
  • 2. Adding/removing HTML elements
  • 3. Calling/executing JS functions

NOTE: The scope of HMTL nodes SootheApp has access to is configurable

How does Client-side routing work?

SootheApp has routing functions that can be manually called as well as the ability to integrates with your browser tab history. This means that SootheApp can react to address changes and other navigation events to trigger actions via a middleware pipeline.

The middleware pipeline uses regex patterns to map middleware functions to routes and will be matched and executed in sequential order. A middleware step can choose to either continue running the next middleware or terminate the middleware execution process. This gives developers the ability to run both intermidden services and resolve endpoints as needed.

During routing, route resolution information, route variables, object state, and the SootheApp context are all passed in to each piece of middleware executed. This should help cover any type of use cases needed when routing.

Out-of-the-box, SootheApp comes with a basic configurable view rendering system. Users can extend this system to easily create custom views that can be inserted into the middleware pipeline. This view system allows for easy fetching and aggregation of HTML, JavaScript, and JS data into a single page/route request.

Getting Started

Launching the framework

Getting started is easy. The only file we need is the Soothe JS file. Once you've done this, the following code will create an instance of SootheApp

  • The line of code creates a new instance of the framework. Once instaniated, the framework will immediately begin running. 
    const spa = new SootheApp();
    An empty set or partial set of options can be passed in to change some basic setup of SootheApp 
                                        const options = {};
                                    let spa = new SootheApp(options);

Setting SootheApp options

SootheApp has a few basic options. Bellow are the default options configured 

                            const options = {
                                dom: {
                                    root: "body"
                                },
                                routing: {
                                    hashRouting: true,
                                    exactMatch: true
                                }
                            };
                            const spa = new SootheApp(options);
options.dom represents DOM related configuration
  • options.dom.root: is the HTML selector for what part of the DOM will be listening/respoonding to SootheApp events
options.routing represents routing related configuration
  • options.routing.hashRouting: tells SootheApp to ignore the URL path during routing and only use the URL fragment for resolving routes. If disabled, full URL paths will be used for route matching
  • options.routing.exactMatch: tells SootheApp that all defined routes will be treated as exact matches and not partial ones

Creating Models

Creating models is easy, we can use the following method: 

SootheApp.prototype.bindModel(modelName, data)
  • Params
    • modelName: name to assign to the model so it can be referenced
    • data: the underlying object that will be encapulsated in the model
  • Return: The resulting model being create (a proxy object wrapping the passed in data)

The resulting code would look as follows: 

                            const spa = new SootheApp();
                            const modelName = "state";
                            let data = {};
                            let state = spa.bindModel(modelName, data);

Whenever you bind a model, model events are automatically fired for the model and all of it's nested properties. If you bind a model with the same name of an existing model, the old model will be unbound triggering an unbind event. The new model will then be bound (trigger yet another bind event) and replace the old model. Any references to the old bound model will no longer trigger events when modified.

To unbind a model, we can call either 

SootheApp.prototype.unbindModel(modelName)
or 
SootheApp.prototype.unbindAllModels()

To retrieve already bound models from SootheApp, we can use the following methods: 

  • SootheApp.prototype.getAllModels()
    • Returns: array of objects: 
      [ {name: "modelName", model: proxy} ]
  • SootheApp.prototype.getModel(modelName)
    • Returns: array proxy of the model
NOTE: It's important to note that the model getters listed above support both root model names and the names of their children

Sometimes getting just the model isn't enough and we need access to the raw data underneath it. We can get that two ways:

  • By calling a method to retrieve it 
    SootheApp.prototype.getModelData(modelName)
    • Returns: object of the underlying data for the model
  • By calling the raw data property of the model. This property is configurable with SootheApp.prototype.rawModelDataProperty. The default value is set to __this 
                                        const spa = new SootheApp();
                                    let model = spa.bindModel("state", {});
                                    console.log(model.__this);

Using Models

When we bind a model, we can initialize some starting data in the model. After retrieving the model, we can also alter that data by adding/setting/removing model properties or child properties just like it were a regular object. When editing the proxy, the data object underneath will also change.

Here's an example of using an already bound model 

                            const spa = new SootheApp();
                            const modelName = "shoppingCart";
                            let data = {location: "Walmart Branch 1", items: [], lockCart: true};
                            let model = spa.bindModel(modelName, data);
                            //Change existing property
                            model.location = "Walmart Branch 2";
                            //Added a new property
                            model.couponCode = "BURGERS";
                            //Adds a child property
                            model.items.push(["apple"]);
                            //Removed a property
                            delete model.lockCart;

It is important to notice that when a JS object is bound to a model, the returned result is a proxy object wrapped around your data. However, SootheApp can only detect changes made directly to the proxy object itself. If you modify the underlying data directly without accessing the proxy, SootheApp will not detect the change, and therefore events will not be dispatched.

Using the model proxy model comes with a few warnings:

  • When your JS logic wants to process the data, we always want to edit a copy of the data and not the model itself to avoid mutating the source. The common way to make a copy of an object is serializing and deserializing it. A proxy cannot be serialized, so you must get the underlying data behind the proxy in order to do this. 
                                        let data = {
                                        firstName: "Bob", lastName: "Belcher"
                                    };
                                    const model = spa.bindModel("state", {name: })
                                    //Don't do this, you'll get an error
                                    let copy = JSON.parse(JSON.stringify(model))
                                    //Do this instead
                                    let copy = JSON.parse(JSON.stringify(data))
  • Do not attempt to set a proxy property to another proxy property. Doing a set to the model will assign the underlying value to the proxy reference to that data, therefore the actual data will be lost. If you want to set a proxy property equal to another proxy property, you must get the underlying data of the second property
    An example of this is the following: 
                                        let data = {
                                        firstName: "Bob", lastName: "Belcher"
                                    };
                                    const model = spa.bindModel("state", {name: })
                                    //Don't do this, you'll lose data!
                                    model.firstName = model.firstName
                                    //Do this instead
                                    model.firstName = data.firstName;

Once we've got our intial data hooked up to the app via models, we're reading to start harnessing the power of models. We'll use events to determine when to read data from models and when we need to update them as well.

Model and Property Name Notation

Before responding to events, we must understand model notation, which is how models are represented as strings inside SootheApp. When creating a model, one must use proper JS identifier naming. Outside of this rule, performing look-ups and utilizing SootheApp features have a slightly different notation.

Model notation allows a developer to travel inside the nested structure of a model using your standard "dot" operator you'd use on a JS object. Model notation follows the same rules as traditional JS object identifiers except functions and square braces "[]" are not allowed. If you need to access an array item, you must use the dot operator.

Here are a few examples of SootheApp models and their dot notation 

  •                                 payroll = {
                                    employees: [
                                        {
                                            id: 1,
                                            firstName: "Bob",
                                            lastName: "Belcher",
                                            pay: 145.65
                                        }
                                    ],
                                    accounts: [
                                        {
                                            id: 0,
                                            number: 1234839
                                        }
                                    ]
                                    date: "12/01/21"
                                }
    • payroll
    • payroll.employees
    • payroll.employees.0
    • payroll.employees.0.id
    • payroll.date

In addition to specifying names of nested properties, we can also use up to one asterisk ("*") as a wildcard operator in our model name. This asterisk would replace one of the property names inside our modal notation such as "person.firstName" could become "person.*" or "*.person" when using a wildcard. Alternatively, a single asterisk with no dot operator means any model.

The wildcard operator is useful for when we want to monitor changes to multiple properties with one listener. Using the payroll example we had above, we the following notations could represent the following combinations of properties:

  • payroll.*
    • payroll.employees
    • payroll.accounts
    • payroll.date
  • *.employees
    • payroll.employees
    • walmart.employees (assuming we had another modle called walmart)
  • payroll.employees.*
    • payroll.employees.0
    • payroll.accounts.1
  • payroll.employees.0.*
    • payroll.employees.0.id
    • payroll.employees.o.firstName
  • *
    • payroll
    • payroll.employees
    • payroll.employees.0
    • payroll.employees.0.id
    • walmart (assuming we have a model with this name)

When referring to a property name, we can only use a single identifier name and do not have access to the dot operator. Despite this, we can still use a wildcard operator to refer to any property name. For example, you would represent the following properties below as:

  • payroll.employees.0.firstName
    modelName: payroll.employees.0
    propertyName: firstName
  • payroll.employees.*
    modelName: payroll.employees
    propertyName: *

Using this model and property notation, we can tell our DOM elements and JS callbacks how to subscribe for events as well as perform other operations that make use of model and property names.

Model Events

A model event triggers when SootheApp determines that a change to a model or DOM could have impact to anything monitoring a model.

The following scenarios is a complete list of scenarios that trigger a model event:

  • Notifies only the target HTML element
    • New element added to the DOM
    • A SootheApp related attribute on an HTML element changes
      NOTE: except for template or expression attribute changes
  • Notifies all listening HTML elements and JavaScript listeners
    • New model bound
    • Model unbound
    • Model property or child propety modified (added, changed, or removed)
    • A 2-way bound value changed via the DOM

A SootheModelEvent has the following properties:

  • app: a reference to the instance of the SootheApp that generated the event
  • model: the proxy object of the model triggering the event (this can be a nested object and not the root model)
  • data: the underlying data underneath the model
  • modelName: the path name to the model in question
  • propName: the name of the property
  • previousValue: the previous value of the model property before the event triggered
  • type: string representing the name of the type of event generated
  • target: the source HTML element triggering the event
    NOTE: will always be the SootheApp root element unless a HTML node specifically changes a value
  • varName: the fully qualified name of the property including it's model and propName
  • rootModelName: the name of the root model bound to SootheApp

There are 4 default event types generated in SootheApp

  • bind
    A bind event symbolizes that new data is being made available by a model that was not previously avaliable, or that an HTML element is receiving data from a model that it did not previously have. A bind event happens when the following actions occur:
    • A model is bound via SootheApp.prototype.bindModel
    • When a new property that was not previoulsy set is added to a model or it's child properties
    • When an HTML element is added to the DOM
    • When an HTML element has a change to one of it's SootheApp attributes
  • set
    A set event symbolizes that a new value has been set on an existing model (typically a value different than what was previously stored) A set event happens when the following actions occur:
    • An existing property on a model or one of it's children was set to a new value
  • unbind
    An unbind event symbolizes that a property that was once previously available on a model or once of it's children is no longer present. An unbind event happens when the following actions occur:
    • A model is unbound
    • A parent property value was set to a different value that does not contain the child property
    • A model property was deleted
  • handler
    A handler event symbolizes that an HTML element had an event handler triggered A handler event happens when the following actions occur:
    • An HTML element fires an event handler that is bound to SootheApp

Using these event details, you can customize how your app will respond to these different events using SootheApp HTML element attributes and JS model listeners.

Model listeners

Model listeners work just like event listeners in your typical JS code. We can attach any number of model listeners to SootheApp. These model listeners will match based on a set of options you pass in as a parameter.

Model listener options has the following structure new SootheModelListenerOptions(modelName, propName, events, condition, discrete)

  • modelName: model Name
    • defaults to "*"
  • propName: prop Name
    • defaults to "*"
  • events: array of strings
    • defaults to empty array []
    • empty array means any/all events
  • condition: function(modelEvent) or true/false expression
    • defaults to true
    • A true value means the callback will fire when the event is triggered
    • False means the event will be ignored and the callback will not be triggered
  • discrete: true, false, or undefined
    • defaults to undefined
    • Discrete means if an actual prop is being passed or is it a generic prop (prop name of a wildcard)
    • Undefined means the callback will trigger whether the property name is generic or not
    • True means the callback will only be triggered if the property name is not generic
    • False means the callback will only be triggered if the property name is generic

Once you've decided on your options, you can add a model listener to soothe app using the function

Soothe.prototype.addModelListener(options, callback)
  • Parameters
    • options: SootheModelListenerOptions that determine when the callback is fired
    • callback: (SootheModelEvent) => {} : the callback to process the event
  • Return: SootheModelListener
An example would look like:
                            const spa = new SootheApp();
                            let model = spa.bindModel("person", {firstName: "Bob", lastName: "Belcher"});
                            const listenerOptions = {modelName: "person"};
                            const callback = function(event) {
                                console.log(event);
                            }
                            const listener = spa.addModelListener(listenerOptions, callback);

If you need to retrieve a listener, you have two options to request them

  • SootheApp.prototype.getModelListeners(options)
    • Parameters
      • options: SootheModelListenerOptions object to query against
    • Return: array of model listeners that match the options search criteria
  • SootheApp.prototype.getListenersForModel(modelName)
    • Parameters
      • modelName: name of the model
    • Return: array of model listeners who's model name matches or is a child model of the name
    • example 
                                                      spa.addModelListener({modelName: "person"}, console.log);
                                                spa.addModelListener({modelName: "person.name"}, console.log);
                                                spa.addModelListener({modelName: "person.name.first"}, console.log);
                                                spa.getListenersForModel("person.name");
                                                //Returns model for person.name and person.name.first

In the event you need to remove a model listener, you can call the following methods

  • SootheApp.prototype.removeModelListener(listener)
    Parameters
    • listener: the SootheModelListener to be removed
  • SootheApp.prototype.removeAllModelListeners()
